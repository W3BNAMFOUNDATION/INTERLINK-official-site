INTERLINK™ — Ingest Engine

Capture, process, and transform every source into structured intelligence.

The ingest engine is the input layer of the sovereign INTERLINK™ stack, turning documents, media, and live feeds into semantic chunks enriched with metadata and vector embeddings.

Built-in ingestion types

  • ✓ PDFs with multi-column reading and OCR fallback
  • ✓ URLs and webpages with link sanitization and clutter removal
  • ✓ YouTube transcripts with timestamped chunking and language selection
  • ✓ Local files: TXT, MD, HTML, RTF, CSV, JSON, and logs
  • ✓ Images with OCR, transcripts, and raw data streams
  • ✓ API feeds and offline, air-gapped ingestion

Outputs

  • • Semantic chunks optimized for retrieval
  • • Metadata-rich knowledge packets
  • • Structured memory entries
  • • Searchable vector embeddings

Autonomous jobs

Schedule ingestion runs, periodic scraping, batch PDF processing, and enrichment campaigns. Pipelines can be triggered manually, via agents, from the CLI, or fully offline.

INTERLINK™ — Route-Core

Router & Messaging Layer for deterministic, multi-service orchestration.

Route-Core is the communication backbone that normalizes every message, selects the right model or agent, and streams unified responses to the frontend, CLI, or API consumers.

Responsibilities

  • ✓ Parse text, files, instructions, and agent commands.
  • ✓ Normalize messages into a universal schema.
  • ✓ Dispatch to Ollama, DeepSeek, Groq, or internal agents via service profiles.
  • ✓ Stream token-by-token responses over WebSockets.
  • ✓ Return uniform output payloads with status and token counts.

Universal message format

Every module speaks the same envelope to avoid duplication and circular imports.

{
  "type": "text",
  "role": "user",
  "content": "Hello world",
  "service": "deepseek"
}

Responses return { status, tokens, content } in the same structure across REST and WebSocket clients.

Routing table

Declarative paths keep API, ingest, memory, and streaming endpoints consistent.

  • • /api/send → Route-Core → Service Profiles → AI Engine
  • • /api/ingest → Ingest Engine
  • • /api/memory/search → Memory Engine
  • • /ws → WebSocket stream router
  • • /cli → CLI integration layer

Service profiles

Requests map to declarative profiles that choose the correct client and model.

services:
  ollama:
    type: "ollama"
    base_url: "http://localhost:11434"
    model: "llama3.2"

  deepseek:
    type: "openai_compat"
    base_url: "https://api.deepseek.com/v1"
    env_key: "DEEPSEEK_API_KEY"
    model: "deepseek-chat"

WebSocket streaming

Keeps the frontend, CLI, and external consumers in sync at ws://localhost:8000/ws with token-by-token updates.

Security guardrails

  • ✓ JSON validation and service whitelisting.
  • ✓ Rate limiting and token caps per request.
  • ✓ Output sanitization for deterministic responses.

Deterministic behaviour

Route-Core enforces separation of responsibilities, model selection per request, and consistent transports so INTERLINK™ behaves like one cohesive fabric.

INTERLINK™ — FS-Core

Secure Filesystem Layer for sandboxed, deterministic access.

FS-Core is the hardened I/O foundation for every INTERLINK™ module. It normalizes paths, blocks traversal, enforces file policies, and keeps agent sandboxes isolated across Windows, Linux, and Mac.

Core workspace layout

Only these namespaced paths are accessible to agents, memory jobs, automation, and uploads.

/workspace/
  /ingest/
  /memory/
  /tmp/
  /agents/
  /uploads/
  /automation/
FolderPurpose
ingest/Raw files entering the ingest pipeline.
memory/Long-term JSONL shards and vector indexes.
tmp/Ephemeral work files cleared on each task.
agents/Namespaced agent working directories.
uploads/Validated user uploads with type checks.
automation/Task manifests, triggers, and scripted jobs.

Security architecture

  • ✓ Path normalization with traversal blocking for every read/write.
  • ✓ Whitelisted base paths limited to /workspace/*.
  • ✓ Sandboxed execution with per-agent isolation and scoped quotas.
  • ✓ File-type validation that rejects executables (.exe, .sh, .ps1, .dll, .bin, .so).
  • ✓ Automatic cleanup for /workspace/tmp/ on each task start.
  • ✓ Deterministic folder layout to guarantee reproducible paths.
Offline and air-gapped deployments require no external dependencies or cloud access.

Capabilities

  • • Safe reads/writes with enforced size limits and normalization.
  • • Directory creation, listing, and deletion within whitelisted roots.
  • • Controlled uploads with MIME validation.
  • • Sandbox resets to clear temp files and restore a clean state.

Sample operations

Deterministic commands used by agents, ingest jobs, and automation scripts.

fs.write("tmp/test.txt", "Hello INTERLINK")
content = fs.read("tmp/test.txt")
items = fs.list("memory/")
fs.mkdir("agents/agent7/")
fs.delete("tmp/task-cache.txt")
fs.reset()

INTERLINK™ — AutoFlow-Core

Deterministic automation engine for schedules, triggers, and multi-step workflows.

AutoFlow-Core runs your scheduled jobs, background tasks, and multi-agent automations with full auditability and offline safety.

What it powers

  • ✓ Scheduled tasks and interval jobs
  • ✓ Background automations with deterministic ordering
  • ✓ File, event, and time-based triggers
  • ✓ Multi-step workflows across ingest, memory, and agents
  • ✓ Fully offline sovereign operation

Workspace layout

/workspace/automation/
  tasks/
  triggers/
  logs/
  manifests/

Tasks execute inside the sandboxed /workspace/automation/ namespace for predictable, repeatable runs.

Task manifest example

id: "task_ingest_sync"
description: "Ingest PDF and store memory"
steps:
  - action: ingest_file
    file: "ingest/report.pdf"
  - action: embed
    model: "default"
  - action: memory_store
    target: "memory/shards/"

Supported actions include ingest_file, embed, memory_store, fs_write, fs_delete, cli_command, agent_run, and optional webhook_post.

Trigger & schedule options

  • • Cron-style schedules (0 */6 * * *) and interval timers (15m).
  • • File and directory watchers for new uploads or changes.
  • • Event triggers like AGENT_FINISHED to chain workflows.
  • • On-demand execution via CLI or API for instant runs.

Each trigger references a manifest ID to launch the correct deterministic pipeline.

Security & isolation

  • ✓ Sandboxed execution with per-task runtime limits and rate caps.
  • ✓ No shell or network access unless explicitly enabled.
  • ✓ Memory isolation across tasks with deterministic cleanup.

Audit logging

Workspace quickstart

  1. Install the desktop app and sign in with your workspace key.
  2. Create your first project and select your preferred LLM stack (online/offline).
  3. Connect storage: choose local embeddings or your enterprise vector database.
  4. Import a PDF, URL, or repository into memory and start chatting with the context.
